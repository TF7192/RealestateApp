## Executive Summary

**Overall posture**: the app is **functional and shippable** today, but **not yet hardened for paying customers at scale**. The architecture is sound (Fastify + Prisma + React 19 + Capacitor), feature coverage is wide, and the recent sprints have closed major feature gaps. The outstanding risks are concentrated in four places:

1. **Security — 2 critical + 2 high**: `JWT_SECRET` falls back to a hardcoded dev value in production, `@fastify/jwt` and `fastify` carry unpatched CVEs, no global 401 handler, no per-endpoint rate limiting on auth.
2. **Error handling — 2 high**: no React error boundary at the root, session expiry silently dumps users, no retry/timeout in the API layer, no offline draft persistence.
3. **Performance — 2 high**: 548 KB main bundle, no list virtualization on lists that will grow past 200 items, no pagination (everything loaded into memory).
4. **Testing — 2 high**: no unit tests, no E2E tests, no pre-deploy test gate in CI — broken code reaches production only to be caught by a 5-second health-check window.

**Total findings**: 128 (pre-dedup) → **~95 discrete findings** (after cross-section dedup, see Section Zero).

**Severity mix** (deduped):
- Critical: 3
- High: 11
- Medium: 40
- Low: 41

**Recommended triage**: the 14 Critical/High items unblock the move from "works" to "safe to sell". Most are S–M effort. Medium items are the meat of the polish pass; Low items are follow-up hygiene.

## Section 0 — How this doc is organized

Each finding below uses the format mandated by the brief:

```
## Finding: [title]
**Where:** [file:line or screen]
**Severity:** Critical / High / Medium / Low
**User impact:** [what the user experiences]
**Root cause:** [technical]
**Proposed fix:** [what to change]
**Effort:** S / M / L
```

Severity is calibrated to **paying-customer impact**, not developer effort:

- **Critical** — data loss, account takeover, security hole, or app entirely broken for a visible fraction of users.
- **High** — regular-path regression, crash-class UX bug, silent data quality degradation, or compliance gap.
- **Medium** — polish gap a customer would mention in feedback; doesn't block the sale but erodes trust over time.
- **Low** — cosmetic, consistency, dev-experience, or nice-to-have.

Effort is a rough t-shirt: **S ≤ 2h**, **M ≤ 1 day**, **L = multi-day**.

Cross-section duplicates are resolved by keeping the finding in its most natural section and cross-referencing the others with `(see F-N.M)`.

## Section 2 — Error Handling & Recovery

### F-2.1: No React error boundary at app root
**Where:** `frontend/src/App.jsx` — no `componentDidCatch` / `getDerivedStateFromError` anywhere
**Severity:** High
**User impact:** A single render crash in any page (e.g. PropertyDetail with malformed data) takes down the whole app — user sees a blank white screen, must close and reopen.
**Root cause:** No boundary wraps `<AppRoutes />`.
**Proposed fix:** Add a `RootErrorBoundary` that catches, logs to analytics, and renders a friendly Hebrew recovery UI ("משהו השתבש · רענן / חזור לדשבורד / דווח"). Also add per-route boundaries around Layout's Outlet so one area can crash without killing navigation.
**Effort:** M

### F-2.2: 401 responses don't trigger a global logout → relogin flow
**Where:** `frontend/src/lib/api.js:50-54`
**Severity:** High
**User impact:** Token expires → first next click returns 401 → generic error toast → user is stuck, unsure if they should refresh. No `return_to` param so they lose their place.
**Root cause:** `request()` throws an Error with `.status = 401`, but nothing in `auth.jsx` listens for 401 to clear state.
**Proposed fix:** In `request()`, when `res.status === 401` AND the user was authenticated, clear auth cookie+state and navigate to `/login?from=<pathname>`. After login, bounce back.
**Effort:** M

### F-2.3: API layer has no retry and no timeout
**Where:** `frontend/src/lib/api.js:46`
**Severity:** High
**User impact:** A 2-second WiFi blip kills a property save — user sees "הטעינה נכשלה", retypes everything. A hung fetch (backend stall) never resolves; user waits forever.
**Root cause:** Raw `fetch()` with no `AbortController`, no retry loop. `request()` surfaces every failure identically.
**Proposed fix:** Wrap `fetch()` with `AbortController` (20 s default timeout for GET, 60 s for POST). For idempotent GET requests, retry on network error / 502 / 503 / 504 with exponential backoff (3 attempts, 200 / 400 / 800 ms). Typed error classes so UI can branch on `NetworkError` vs `ServerError`.
**Effort:** M

### F-2.4: Error messages leak raw HTTP status codes to users
**Where:** `frontend/src/lib/api.js:51` — `HTTP ${res.status}` fallback
**Severity:** Medium
**User impact:** Agent sees `HTTP 502 Bad Gateway` instead of "לא ניתן להתחבר לשרת כרגע — נסה שוב בעוד רגע."
**Root cause:** Server doesn't always return a Hebrew `error.message`; client falls back to English HTTP shorthand.
**Proposed fix:** Every backend error response is `{ error: { message: <Hebrew>, code: <machine-readable> } }`. Client fallback becomes "משהו השתבש — נסה שוב." (never HTTP NN).
**Effort:** S

### F-2.5: No offline detection / banner
**Where:** `frontend/src/components/OfflineBanner.jsx` exists but isn't fed by any online-detection hook
**Severity:** Medium
**User impact:** User types into a property form while offline, hits save, gets a cryptic network error and loses the edits.
**Root cause:** OfflineBanner is a static component. No `navigator.onLine` subscription, no mutating-writes guard.
**Proposed fix:** Add `useOnline()` hook (listen to `online`/`offline` events + ping `/api/health` every 30 s). OfflineBanner consumes it. Writable forms grey out their primary CTA while offline AND autosave to localStorage every 5 s so nothing is lost. On reconnect, the banner changes to "חיבור חזר — שומר שינויים…".
**Effort:** M

### F-2.6: 404 is a silent redirect to `/`
**Where:** `frontend/src/App.jsx:174` — `<Route path="*" element={<Navigate to="/" replace />} />`
**Severity:** Medium
**User impact:** A mistyped URL or an old bookmark silently dumps the user on the dashboard, no explanation.
**Root cause:** Catch-all uses Navigate.
**Proposed fix:** `NotFoundPage` component — friendly Hebrew copy, a "חזור לנכסים" CTA, a secondary "חזור אחורה" button, and a search input that routes to `/properties?search=`.
**Effort:** S

### F-2.7: Detail pages (property/lead/owner by id) have no "not found" state
**Where:** `PropertyDetail.jsx`, `CustomerDetail.jsx`, `OwnerDetail.jsx`
**Severity:** Medium
**User impact:** Passing a bad id in the URL renders a broken, half-rendered page with console errors rather than a clean "לא נמצא" screen.
**Root cause:** Pages assume `property/lead/owner` is truthy after `await api.getX(id)`.
**Proposed fix:** On 404 from the API, render a typed NotFoundPanel with "הנכס/ליד/בעל נכס נמחק או לא קיים" + "חזור לרשימה".
**Effort:** S

### F-2.8: No stale-data refetch on tab refocus
**Where:** `Properties.jsx`, `Customers.jsx`, `Dashboard.jsx` — no `visibilitychange` subscription
**Severity:** Low
**User impact:** User leaves a tab open overnight; returns to see last week's prices.
**Root cause:** Components fetch only on mount.
**Proposed fix:** A small `useRefreshOnRefocus()` hook that refetches when `document.hidden` flips from true to false AND last fetch was >5 minutes ago.
**Effort:** S

## Section 3 — Loading States

### F-3.1: Lists and detail pages show spinners or nothing, not skeletons that match final layout
**Where:** `Properties.jsx:835-844`, `Customers.jsx:304-331`, `Dashboard.jsx:84` (all use `useDelayedFlag`); `PropertyDetail.jsx` uses the "טוען…" string
**Severity:** Medium
**User impact:** On slow network (3G, hotel WiFi), there's 500 ms of nothing then a pop-in of the list — perceived as a broken page.
**Root cause:** `useDelayedFlag` waits 220 ms before showing a skeleton, which is correct to prevent a flash on fast LAN but wrong threshold for slow mobile. Skeletons also don't match actual card heights → layout shift.
**Proposed fix:** Replace delay-then-skeleton with skeleton-from-first-frame-but-fade-in-at-100ms. Match skeleton height to actual card metrics to avoid CLS. Add real skeletons to the 5 screens above (currently only a couple have them).
**Effort:** M

### F-3.2: Optimistic updates are used in one place only
**Where:** `frontend/src/lib/toast.jsx` exports `optimisticUpdate(...)` helper — grep shows limited usage.
**Severity:** Medium
**User impact:** Toggling a property's "active" status or a lead's "hot/cold" status waits for the server roundtrip → perceptible lag on every tap.
**Root cause:** Helper exists, is barely adopted.
**Proposed fix:** Audit every toggle-like mutation; convert to optimistic-with-rollback where destructive consequences are low (status toggles, pick/unpick chips, favorite). Never optimistic for delete/create.
**Effort:** M

### F-3.3: Video uploads have an `onProgress` hook but the UI doesn't render progress
**Where:** `frontend/src/lib/api.js:155-182` — `uploadPropertyVideo` accepts `onProgress`; `PropertyPhotoManager.jsx` doesn't display a percentage
**Severity:** Low
**User impact:** A 30 MB video upload shows a spinner for 60 s with no indication of progress. Users cancel.
**Root cause:** Wiring was 80% done and never finished.
**Proposed fix:** Add `<progress max={100} value={progress} />` inside `PropertyPhotoManager`'s upload tile. Also add a cancel button.
**Effort:** S

### F-3.4: Search inputs are inconsistently debounced
**Where:** `AdminUsers.jsx:28-73` (300 ms); `AddressField.jsx:50-101` (clearTimeout + setTimeout ad-hoc); `SmartFields.jsx` (350 ms); most page-level search inputs (`Properties.jsx`, `Customers.jsx`) are **not debounced at all**
**Severity:** Medium
**User impact:** Typing "ירושלים" in the Customers search fires 6 re-renders in a row; on a 500-row list that's 3000 component renders.
**Root cause:** No shared debounce hook — each component rolls its own or skips it.
**Proposed fix:** Ship a `useDebouncedValue(value, ms)` hook in `lib/`, apply 300 ms globally on text filters and 200 ms on autocomplete.
**Effort:** S

### F-3.5: Dashboard blocks on the slowest of 4 parallel calls
**Where:** `Dashboard.jsx:84`
**Severity:** Medium
**User impact:** If the owners-count endpoint is slow, the agent sees a blank dashboard even though properties + leads are already loaded.
**Root cause:** Page waits for all 4 endpoints before rendering anything.
**Proposed fix:** Render the welcome header + skeleton tiles from frame 1. Each KPI tile owns its own fetch and loading state — one slow tile doesn't hold the others.
**Effort:** M

## Section 5 — Forms: Validation, Save, Recovery

### F-5.1: Validation fires immediately, not on blur
**Where:** Several forms compute errors eagerly; no `touched` tracking
**Severity:** Medium
**User impact:** The moment the form renders, required fields show red "חובה" — the user feels scolded before typing anything.
**Root cause:** No per-field `touched` state.
**Proposed fix:** Introduce a tiny `useFieldTouched(key)` hook. Render field-level errors only when `touched[key] && error[key]`. Submit button still disabled on any error, but without the red outline on untouched fields.
**Effort:** M

### F-5.2: Save patterns aren't consistent across forms
**Where:** `NewProperty.jsx` (explicit save step 1 + autosave to draft), `NewLead.jsx` (autosave), `Profile.jsx` (explicit save), `QuickEditDrawer.jsx` (explicit)
**Severity:** Medium
**User impact:** Agents aren't sure whether their changes auto-save. In Profile you have to press save; in NewLead you don't; in NewProperty you sometimes do, sometimes don't.
**Root cause:** No design spec; each form chose its own pattern.
**Proposed fix:** Document the rule: **forms that create a resource** require explicit Save; **forms that edit an existing resource with a single field/section** autosave with a visible "נשמר" indicator. Refactor outliers to match.
**Effort:** M

### F-5.3: NewProperty step 2 data is lost if the user goes back to step 1 in create mode
**Where:** `NewProperty.jsx:590-612` (goToStep autosaves only in edit mode)
**Severity:** High
**User impact:** Agent fills step 1, advances to step 2, adds photos and features, navigates back to fix a typo in step 1 → returns to step 2 and every photo / checkbox / free-text is blank.
**Root cause:** In create mode, `goToStep` doesn't persist step 2 to the draft.
**Proposed fix:** Autosave the full form state to the draft on every `goToStep` regardless of mode. Clear the draft only on final create-success.
**Effort:** S

### F-5.4: Save failures don't preserve the form
**Where:** `NewProperty.jsx:517-551` (both step catches just set an error string)
**Severity:** High
**User impact:** POST /properties fails → toast says "שמירה נכשלה" → user presses refresh / navigates away → everything typed is gone.
**Root cause:** `catch` doesn't flush the current state to the draft.
**Proposed fix:** In catch blocks, call `flushDraft(form)` synchronously before rendering the error; the recovery banner on next page-load will offer "המשך טיוטה".
**Effort:** S

### F-5.5: No `beforeunload` guard on dirty forms
**Where:** every form page
**Severity:** Medium
**User impact:** Agent accidentally closes a tab mid-edit — no warning, no recovery prompt.
**Root cause:** Nobody listens for `beforeunload`.
**Proposed fix:** `useBeforeUnload(isDirty)` hook. Plus a router-level guard (react-router `useBlocker`) for SPA navigation.
**Effort:** M

### F-5.6: Draft autosave is create-only
**Where:** `NewProperty.jsx:177-193` — `isEdit ? null : { ... }`
**Severity:** Low
**User impact:** Interrupted edits are unrecoverable.
**Root cause:** Explicit guard to avoid confusing "unsaved draft" vs server state.
**Proposed fix:** Separate draft key per edit target (`estia-draft:edit-property:<id>`); on mount offer "יש שינויים שלא נשמרו — שחזר?" banner.
**Effort:** M

### F-5.7: Keyboard nav: Enter inside textareas can accidentally submit
**Where:** assorted form submit handlers
**Severity:** Low
**User impact:** In textareas, Enter should add a line; in single-line inputs, it should submit. Current behavior is inconsistent.
**Root cause:** Inconsistent `onKeyDown` handling.
**Proposed fix:** Audit every form: single-line inputs get `enterKeyHint="go"` + form `onSubmit`; textareas do nothing special with Enter; modifier Enter (Ctrl/Cmd+Enter) submits from a textarea where appropriate.
**Effort:** S

## Section 6 — Navigation & IA

### F-6.1: No breadcrumb on detail pages
**Where:** `PropertyDetail.jsx`, `CustomerDetail.jsx`, `OwnerDetail.jsx`, `Transfers.jsx`
**Severity:** Medium
**User impact:** Arriving at a detail page from search gives you one "back to list" chip and nothing else — you can't tell if you came from filters, from dashboard, or from a deep link.
**Root cause:** No breadcrumb component.
**Proposed fix:** `Breadcrumbs` component: `Home › נכסים › <address>` on PropertyDetail, same pattern for customers/owners. Clickable crumbs restore filter state via URL params (see F-6.5).
**Effort:** M

### F-6.2: Active NavLink state isn't obvious on mobile tab bar
**Where:** `MobileTabBar.jsx` — gold dot is 4×4 px, easy to miss
**Severity:** Low
**User impact:** Agents occasionally tap the current tab thinking they're tapping a different one.
**Root cause:** Subtle design.
**Proposed fix:** Keep the gold dot but also bump the icon stroke weight (already done) and give the active label slightly darker color with slightly heavier weight (already done) — just verify on the 4 new tabs (Properties / Customers / Owners / Calculator) that it's visible.
**Effort:** S

### F-6.3: No scroll restoration on back navigation
**Where:** `Properties.jsx`, `Customers.jsx`
**Severity:** Medium
**User impact:** Scroll 50 rows down, open a property, hit back → list is at the top, have to re-scroll. For a 500-property list this is a meaningful time sink.
**Root cause:** React Router v7 `<ScrollRestoration />` not mounted.
**Proposed fix:** Add `<ScrollRestoration />` to the root route; verify it works with virtualization.
**Effort:** S

### F-6.4: Modals don't trap focus
**Where:** `OwnerPicker.jsx:155`, `QuickEditDrawer.jsx`, `ProspectDialog.jsx`, `LeadMeetingDialog.jsx`
**Severity:** High
**User impact:** Keyboard user opens a modal, tabs out of it, focus lands on a page element behind the backdrop — they can activate hidden UI without realizing.
**Root cause:** `role="dialog"` set but no `aria-modal="true"` on every dialog, no focus trap.
**Proposed fix:** A `<Dialog>` wrapper that (a) applies `role="dialog"` + `aria-modal="true"`, (b) traps Tab focus inside the dialog, (c) restores focus to the trigger on close. Use everywhere.
**Effort:** M

### F-6.5: Filters/sorts aren't in the URL
**Where:** `Properties.jsx` imports `useSearchParams` but never applies advFilters to it
**Severity:** High
**User impact:** Agent filters to "Tel Aviv, 3+ rooms, 2-3M ILS", copies the URL to send to a colleague, colleague opens → sees an unfiltered list. No shareable saved searches.
**Root cause:** Filter state is local-only.
**Proposed fix:** Serialize filters to query string (`?city=tel-aviv&minRooms=3&minPrice=2000000`). On mount, hydrate filters from URL. Every filter change writes back.
**Effort:** M

### F-6.6: Pressing Escape closes a modal but leaves the page scrolled where it was dragged
**Where:** various modals
**Severity:** Low
**User impact:** Minor — sometimes the page has drifted while the modal was open.
**Root cause:** No scroll snapshot.
**Proposed fix:** `<Dialog>` wrapper snapshots scrollY on open, restores on close.
**Effort:** S

## Section 7 — Visual Consistency

### F-7.1: Hex color literals scattered across page-level CSS
**Where:** `ProspectSign.css` (`#f8f4ea`, `#1e1a14`, `#6b6356`), `Yad2Import.css` (`#1a1409`), `Templates.css` (`#2a2a2e`, `#0e7b66`) — ~30+ hits

**User impact:** Color changes or theme variants (dark mode, accessibility high-contrast) require hunt-and-replace; dark mode is inconsistent (F-7.5).
**Root cause:** Features shipped before the token layer in `index.css` was canonical.
**Proposed fix:** Canonical token set in `index.css`. Sweep all page-level CSS, replacing hexes with `var(--color-*)`. New hexes blocked via a stylelint rule.

### F-7.2: Login buttons use custom classes instead of the `.btn` system
**Where:** `Login.jsx:163-176` — `.auth-method-btn` / `.phone-btn` / `.signup-btn`
**Severity:** Low
**User impact:** Login page looks subtly off-brand vs the rest of the app.
**Root cause:** Login was designed in isolation.
**Proposed fix:** Unify to `.btn btn-primary` / `.btn btn-secondary` / `.btn btn-ghost`.
**Effort:** S

### F-7.3: 18+ files still use physical (non-logical) margin/padding
**Where:** `Profile.css`, `Customers.css`, `Properties.css`, `Layout.css`, `MobilePickers.css`, etc.
**Severity:** Low
**User impact:** RTL rendering is held together by `direction: rtl` on `<html>`; any LTR future work or text-direction toggle will flip layouts.
**Root cause:** Pre-logical-property era code.
**Proposed fix:** Migrate to `margin-inline-start` / `padding-inline-end` etc. Do it file-by-file as you touch them, not as one megacommit.
**Effort:** L (distributed)

### F-7.4: Dark mode is declared but incomplete
**Where:** `index.css:62-96` defines dark tokens; `Profile.jsx:279-299` has a toggle; several page CSS files don't override backgrounds in dark.

## Section 8 — Performance

### F-8.1: Main bundle is 548 KB — Lighthouse Performance ceiling ~75
**Where:** `frontend/dist/assets/index-*.js`
**Severity:** High
**User impact:** First-time load on a mid-range phone over 4G is ~4 s to interactive.
**Root cause:** Most pages eagerly bundled; only a few are lazy. Core reducers, all lucide icons, and all shared utilities land in `index`.
**Proposed fix:** Lazy-load every non-default route through `React.lazy`. Split `lucide-react` via the `createLucideIcon` approach it already supports (one file per icon). Audit `api-*.js` (184 KB) for unused exports.
**Effort:** M

### F-8.2: No list virtualization
**Where:** `Customers.jsx`, `Properties.jsx`, `Leads.jsx`, `Owners.jsx`
**Severity:** High
**User impact:** A 500-row list renders 500 DOM nodes. Scrolling is visibly janky on older phones; search input lags (see F-3.4).
**Root cause:** No virtualization library in use.
**Proposed fix:** Add `@tanstack/react-virtual` (lightweight, no context provider). Apply to lists when length > 100. Use the same row height throughout so measurement is cheap.
**Effort:** M

### F-8.3: No pagination on the backend list endpoints
**Where:** `backend/src/routes/properties.ts`, `leads.ts`, `owners.ts` — all return the full result set
**Severity:** High
**User impact:** An agent with 2000 customers downloads 2 MB of JSON on every navigation to `/customers`. Server RAM spikes during load. Frontend can't virtualize what it doesn't know is lazy.
**Root cause:** No `limit` / `offset` / `cursor` params defined on the list routes.
**Proposed fix:** Cursor-based pagination (`cursor` = createdAt+id). Default page size 50; `limit` max 200. Client hydrates lazily via `onEndReached`.
**Effort:** L

### F-8.4: Image lazy-loading is hit-and-miss
**Where:** `loading="lazy"` present on ~3 components (Yad2Import thumbnails, CustomerPropertyView hero, PropertyDetail carousel); absent on Properties cards, Owners cards, Dashboard recent-activity thumbnails
**Severity:** Medium
**User impact:** The properties grid fetches every image at once even for cards below the fold. On a 500-property view with 500 thumbnails, that's 500 × ~80 KB = ~40 MB of image traffic.
**Root cause:** Inconsistent attribute application.
**Proposed fix:** Add `loading="lazy"` + `decoding="async"` to every `<img>` in list-rendered code. Preload only the first visible image of a carousel.
**Effort:** S

### F-8.5: No memoization on list cards → whole list re-renders on any state change
**Where:** `Customers.jsx`, `Properties.jsx` — card components are inline and receive changing props on every render
**Severity:** Medium
**User impact:** Typing in the search box on Customers triggers a 500-card re-render per keystroke. Feels laggy.
**Root cause:** No `React.memo`, no stable callbacks.
**Proposed fix:** Extract `PropertyCard` / `CustomerCard` to memoized components; wrap onClick/WhatsApp handlers in `useCallback` with stable deps. Combine with F-3.4 (debounce) and F-8.2 (virtualize) — they compound.
**Effort:** M

### F-8.6: Lazy-imports ineffective on two Capacitor modules
**Where:** build warnings `INEFFECTIVE_DYNAMIC_IMPORT` on `@capacitor/browser` and `@capacitor/app`
**Severity:** Low
**User impact:** Bundle slightly larger than it should be.
**Root cause:** Static `import` in `src/native/share.js` defeats the dynamic `import()` elsewhere.
**Proposed fix:** Make share.js also dynamic-import, OR drop the dynamic-import in App.jsx (we always need these anyway on native). Pick one.
**Effort:** S

## Section 11 — Security

### F-11.1: `JWT_SECRET` has a hardcoded development fallback
**Where:** `backend/src/server.ts:77-78` → `process.env.JWT_SECRET || 'dev-only-change-me'`
**Severity:** Critical
**User impact:** If the production deploy is ever pushed without the env var set, every JWT is signed with a publicly-knowable secret → trivial session forgery across every account.
**Root cause:** Convenience fallback for dev.
**Proposed fix:** Hard-fail at process start if `NODE_ENV === 'production' && !JWT_SECRET`. Log the error, `process.exit(1)`.
**Effort:** S

### F-11.2: `@fastify/jwt` and `fastify` have open CVEs
**Where:** `backend/package.json` — `@fastify/jwt ≤9.0.1`, `fastify ≤5.8.2`
**Severity:** Critical
**User impact:** Documented fast-jwt cache confusion / algorithm confusion / ReDoS, and Fastify unbounded allocation + Content-Type header bypass.
**Root cause:** Stale pin.
**Proposed fix:** Upgrade `@fastify/jwt` to 10.x (breaking — audit usage), `fastify` to ≥5.8.5 (minor). Rerun `npm audit`.
**Effort:** S

### F-11.3: Auth cookies are unsigned
**Where:** `backend/src/server.ts:77-78` cookie plugin `{ signed: false }` implied / default
**Severity:** High
**User impact:** With unsigned cookies + predictable shape, manipulation is a step easier for an attacker in possession of the network path.
**Root cause:** Simplicity at setup.
**Proposed fix:** Generate `COOKIE_SECRET`, enable signed cookies, store it in the same Secrets-Manager path as `JWT_SECRET`. Re-hash existing sessions on deploy.
**Effort:** M

### F-11.4: No rate limiting on `/auth/login` and `/auth/signup`
**Where:** `backend/src/server.ts:80-84` — one global limit of 300 req/min
**Severity:** High
**User impact:** Brute-force window open on login; spam-register window open on signup.
**Root cause:** Global config.
**Proposed fix:** Per-route limits via `{ config: { rateLimit: { max: 10, timeWindow: '15 minutes' } } }` on auth routes. Stricter for signup (1/min/IP).
**Effort:** M

### F-11.5: 401 responses don't trigger a global logout flow (see F-2.2)
Already captured under Error Handling.

### F-11.6: Helmet CSP disabled
**Where:** `backend/src/server.ts:59` — `await app.register(helmet, { contentSecurityPolicy: false })`
**Severity:** Medium
**User impact:** One XSS vulnerability anywhere in a dep → full privilege escalation in the SPA context.
**Root cause:** CSP friction during early dev.
**Proposed fix:** Enable a minimal CSP: `script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https: blob:; connect-src 'self' https://estia.tripzio.xyz https://www.google-analytics.com`; then iterate. Capacitor WKWebView needs special handling.
**Effort:** M

### F-11.7: File uploads accept any content-type
**Where:** `backend/src/server.ts:86` — `multipart { limits: { fileSize: 100 * 1024 * 1024 } }`
**Severity:** Medium
**User impact:** An attacker can upload `.exe`, HTML, SVG-with-script. SVG served from `/uploads/*` would run in the context of `estia.tripzio.xyz`.
**Root cause:** No whitelist.
**Proposed fix:** Allow `image/jpeg | png | webp | heic`; `video/mp4 | quicktime`; `application/pdf` for exclusivity-agreement. Reject others with 415.
**Effort:** M

### F-11.8: No CSRF tokens; reliance on SameSite cookies
**Where:** `backend/src/routes/*` — all state-changing routes are cookie-auth
**Severity:** Medium
**User impact:** If SameSite policy ever slips (browser update, new iOS Safari behavior), cross-site POST attacks land.
**Root cause:** No defense in depth.
**Proposed fix:** Add a CSRF token for non-GET routes (Fastify has `@fastify/csrf-protection`). Client reads token from a `/auth/csrf` endpoint once per session.
**Effort:** M

### F-11.9: Public agent pages leak existence via 404
**Where:** `backend/src/routes/public.ts:54-60`
**Severity:** Low
**User impact:** Attacker can enumerate agent slugs.
**Root cause:** Response differs between "never existed" and "exists but private".
**Proposed fix:** Same 404 body + timing on both states.
**Effort:** S
