# Security Audit Summary — Estia

Generated by extended whitebox security audit on 2026-04-25 against
`main @ f621a5d`.

## Overall Risk Rating

**Medium.**

No unauthenticated path to account takeover or admin compromise was
found in the current code on `main`. The most pressing item is one
**High**: a brute-force gap on the password-reset endpoint. The rest
are Medium/Low. The application has clearly been hardened — Helmet
CSP, signed `HttpOnly+Secure+SameSite=Lax` cookies, a real `ADMIN`
role enum, agent-scoped Prisma reads, an SSRF allowlist on the Yad2
image rehoster, an SVG-rejecting upload guard, a CSV-injection guard,
and an open-redirect filter on the OAuth callback are all present
and verified.

## Audit Scope

- All 78 backend TypeScript files in `backend/src/` (routes,
  middleware, lib, scripts).
- All 231 frontend JS/JSX files in `frontend/src/`.
- Prisma schema + migrations (`backend/prisma/`).
- Deployment surface: `backend/Dockerfile`, `docker-compose.yml`,
  `docker-compose.prod.yml`, `frontend/nginx.conf`,
  `frontend/capacitor.config.json`, `.github/workflows/*`.
- Dependency manifests + lockfiles for both workspaces (npm audit
  run live during the audit).
- `.env`, `.env.example`, `.gitignore` — verified `.env` files are
  gitignored and only `.env.example` is checked in.

No live testing was performed. All findings come from static review.

## Application Security Posture

The codebase shows a deliberate, defense-in-depth posture:

- **Authentication.** Argon2id password hashing, JWT in
  `HttpOnly+Secure+SameSite=Lax` cookies via `@fastify/cookie`
  signed by `COOKIE_SECRET`, fail-fast on missing
  `JWT_SECRET`/`COOKIE_SECRET` in production, soft-delete check on
  every authenticated request, mock-OAuth route gated by
  `AUTH_ALLOW_MOCK` env, password-reset tokens 24-byte random with
  30-min TTL and not logged at info level.
- **Authorization.** A real `ADMIN` enum on `User.role` with a
  framework-level `app.requireAdmin` decorator. Agent-scoped Prisma
  reads on properties, leads, deals, owners, agreements, contracts,
  documents, AI tools, prospects, public matches, transfers,
  meetings, marketing, market, search, calendar, reminders. Office
  ownership / cross-office transitions guarded.
- **Edge / transport.** Helmet with `default-src 'none'` and
  `frameAncestors 'none'` on the JSON API; nginx terminates with
  HSTS, CSP, X-Frame-Options DENY, X-Content-Type-Options nosniff,
  Referrer-Policy strict-origin-when-cross-origin, Permissions-Policy
  with restrictive defaults. CORS uses an explicit allowlist
  callback, never reflects origin.
- **Storage.** S3 with object-ownership enforcement, presigned 1h
  URLs, ownership-checked download endpoint, `urlToKey` rejects `..`
  and absolute paths.
- **External fetches.** Yad2 image rehoster runs through
  `assertRehostUrlSafe` (allowlist + RFC1918/loopback/link-local
  block); Google/Apple OAuth verify with proper `iss`/`aud`/JWKS;
  token-exchange responses are redacted before logging.
- **Tests.** A meaningful corpus of integration + unit security
  tests landed alongside the hardening: cross-agent agreement
  access, cross-tenant office moves, soft-delete enforcement,
  admin-role gates, neighborhood-admin-only writes, prospect PII,
  CSV cell escaping, OAuth redirect validation, SSRF guard, SVG
  rejection, security headers, document download flow.

The audit nonetheless found one High and a handful of Medium/Low
items below.

## Threat Model Summary

**Likely attackers:** anonymous internet user; a registered AGENT
abusing tenant scoping; a co-agent inside a multi-agent office; an
OWNER abusing office-admin features; a CUSTOMER trying to read
other customers' matches; a third-party carrying a forged OAuth
token; a webhook attacker; a co-resident on a shared deploy.

**Sensitive assets:** brokerage agreements with seller PII +
commission %, prospect signing flows with personal IDs / licenses,
agent address books, S3-hosted documents, AI prompt context,
session JWTs, OAuth refresh tokens (Google Calendar).

**Trust boundaries:** browser ↔ Fastify API; cookie/JWT ↔ Prisma
DB; agent input ↔ AI prompt; user-supplied URLs ↔ outbound
fetches; Capacitor WebView ↔ same-origin prod URL; external OAuth
provider ↔ callback route; deploy worker ↔ EC2.

## Attack Surface Summary

- **Public.** Login/signup/forgot-password/reset-password/contact
  form/landing pages (`/api/public/*`) / OG metadata HTML / sitemap
  / OAuth callbacks / WhatsApp link-preview crawler endpoint.
- **Authenticated agent.** ~40 route modules under `/api/*` for
  the CRM core (properties, leads, owners, deals, agreements,
  documents, calendar, AI, voice, reports, etc.).
- **Owner-only.** `/api/office/*` and `/api/team/*` portions
  gated on `requireOwner`.
- **Admin-only.** Neighborhood + group lookup writes, admin user
  management, AI-usage analytics, chat oversight; all on
  `requireAdmin`.
- **Webhooks/inbound integrations.** None confirmed — no Stripe,
  no provider webhooks; only OAuth callbacks.
- **Native iOS/Android.** Loads the prod URL in WKWebView/
  WebView; cookies host-scoped; Android web-view debugging
  disabled.

## High-Risk Issues

**SEC-001 — `/api/auth/reset-password` brute-force.** The reset
handler at `backend/src/routes/auth.ts:270` accepts `{ token,
password }` but is registered without per-route rate limiting,
unlike its siblings:
- `:224` `/login` → `LOGIN_LIMIT` (10 / 15 min)
- `:211` `/signup` → `SIGNUP_LIMIT` (3 / 1 hour)
- `:246` `/forgot-password` → `FORGOT_LIMIT` (5 / 1 hour)

Tokens are 24 random bytes (192 bits) with a 30-min TTL — the
keyspace is unguessable, but with no per-route brake an attacker
can keep hammering the endpoint over the 30-minute window from a
distributed IP set, and the global 300/min ceiling allows ~9000
attempts inside the TTL. Combined with any side-channel that
narrows the search space (timing, partial leakage, malicious
referer that captures part of the token), the gap becomes real.

The fix is one line — add `{ config: { rateLimit: { max: 10,
timeWindow: '15 minutes' } } }` mirroring `LOGIN_LIMIT`'s shape.

## Medium / Low Issues

**SEC-002 — OAuth flows skip `deletedAt`.** Both
`oauth-google.ts:218` and `oauth-apple.ts:96` do
`prisma.user.findUnique({ where: { googleId | appleId | email } })`
which returns soft-deleted rows; they then mint a JWT and `setCookie`.
The per-request middleware at `middleware/auth.ts:58-62` re-checks
`deletedAt: null` and filters the cookie out, so the user never
actually accesses authenticated routes — but the OAuth callback
completes successfully and the user appears "logged in" until the
first /api/me call 401s. Add an early `if (user.deletedAt) return
reply.code(401)` after the upsert in both files.

**SEC-003 — public lookup leaks non-ACTIVE listings.**
`backend/src/routes/public.ts:113-120` (`GET /agents/:slug/properties/
:propSlug`) and `:233-239` (`GET /lookup/property/:id`) do
`findFirst({ where: { agentId, slug } })` / `findUnique({ where:
{ id } })` with no `status: 'ACTIVE'` predicate. The list endpoint
at `:68` correctly filters by `status: 'ACTIVE'`, so the gap is
detail + lookup only. A property paused/sold/withdrawn but whose
slug survives still resolves through both endpoints, leaking
images / videos / address / sometimes broker-side notes via
`include`.

**SEC-004 — `@xmldom/xmldom` HIGH CVE in frontend deps.**
`npm audit` (frontend) reports:
- `@xmldom/xmldom <0.8.13` — uncontrolled-recursion DoS during XML
  serialization (HIGH).
- `uuid <14.0.0` — buffer bounds check missing (MODERATE).
Both pulled transitively through `exceljs`. Exploitable only when
parsing user-supplied `.xlsx`, in the user's own browser tab —
not a server-side vector — but the High advisory is still worth
clearing. Either upgrade `exceljs` or pin `@xmldom/xmldom` to
≥0.8.13 via npm overrides.

**SEC-005 — OAuth burst rate limits.** `oauth-google.ts:146`
(`/google/callback`), `:299` (`/google/native-exchange`), and
`oauth-apple.ts:59` (`/apple/native-exchange`) lack per-route
rate limits. Apple verification is expensive (JWKS fetch + JWT
verify); the Google native-exchange map is bounded at 10k entries,
so a burst on `consumeNativeCode` can churn it. Add a per-route
limit (e.g. 30 / minute) to all three.

**SEC-006 — Prompt injection in AI prompt builders.**
`backend/src/routes/ai.ts:218` (`/api/ai/describe-property`) and
`:692, 702-705` (`/api/ai/meeting-brief`) interpolate
`property.notes` / `lead.notes` / `meeting.title` / `meeting.location`
/ `meeting.notes` directly into the Claude prompt without
delimiters. In a single-agent context the attacker is the user,
which limits the impact, but the upcoming multi-agent
office + property-transfer flows widen the surface — agent A
poisons a note, agent B's later AI tool follows the injected
directive instead of system policy. Wrap user fields in fenced XML
tags (`<note>…</note>`) and instruct the model to treat tag
contents as data only.

**SEC-007 — `/logout` rate limit missing.** Low impact — a user
can only log themselves out — but adding the same per-route limit
costs nothing.

**SEC-008 — `/api/transfers/agents/search` enumeration.** The
endpoint at `backend/src/routes/transfers.ts:18-45` is rate-limited
to 20/min, but the response shape distinguishes (a) self lookups
(`{ self: true }`), (b) non-existent emails (`{ agent: null }`),
(c) CUSTOMER role (different shape), and (d) found AGENT/OWNER
(returns name + phone). Return the same `{ agent: null }` shape for
non-AGENT-or-OWNER + non-existent + self to neutralize
side-channels.

**SEC-009 — `target="_blank"` without `rel`.** 16 frontend sites
use `<a target="_blank">` without `rel="noopener noreferrer"`,
allowing reverse-tabnabbing. Mass-replace and add the
`react/jsx-no-target-blank` ESLint rule. Files include
`AdvertsPanel.jsx:267`, `AgreementDialog.jsx:166`,
`PropertyAgreementsSection.jsx:220`, `PropertyHero.jsx:235,254`,
`MarketingActionDialog.jsx:210`, `AgentCard.jsx:339,366`,
`pages/Help.jsx:258`, `Profile.jsx:277`, `Documents.jsx:366`,
`Customers.jsx:405`, `Owners.jsx:332`, `PropertyDetail.jsx:1300,
2007`, `Properties.jsx:1389`.

**SEC-010 — yad2 hardcoded admin email.**
`backend/src/routes/yad2.ts:106` keeps a
`Set(['talfuks1234@gmail.com'])` to elevate the Yad2 scrape quota
from 3/hr to 50/hr. Not directly exploitable — it grants more
quota, not auth — but it's the same email-allowlist anti-pattern
that the rest of the codebase has retired in favor of `role ===
'ADMIN'`.

**SEC-011 — Per-process in-memory state.** `yad2.ts` job map,
`market.ts` job map, `import.ts` job map, `oauth-google.ts`
`pendingCodes`, `contact.ts` IP rate-limit buckets all live in
process memory. Fine on the current single-instance EC2; a
correctness issue if/when horizontal scaling lands. `ioredis` is
already a dependency.

**SEC-012 — citext email.** `User.email` is case-sensitive in the
schema; `Adam@…` and `adam@…` can coexist as distinct accounts.
Has not caused incidents. Schema migration tracked.

**SEC-013 — `signed: false` on @fastify/jwt cookie option.**
`backend/src/server.ts:162` registers `@fastify/jwt` with `cookie:
{ cookieName: 'estia_token', signed: false }`. The JWT itself is
HMAC-signed by `JWT_SECRET`, so cookie tampering is detected at
the JWT verify step regardless of the cookie wrapper. This is
defense-in-depth at most. Optional.

## Unknowns / Needs Verification

- **SEC-002 user-impact.** The OAuth callbacks happily upsert into
  a soft-deleted row; the cookie they hand back fails on the next
  request. There is no automated test confirming the user can't
  reach a privileged endpoint with the freshly-issued cookie before
  the middleware re-check fires. Not a security bug given the
  middleware, but worth a regression test.
- **Webhook surface.** No Stripe / SES / external webhook
  endpoints found, but premium gating exists; verify whether the
  premium flag is set by an external provider that should be
  signature-checked or only by manual admin action.

## Final Notes

The application is in good security shape overall — there is no
unauthenticated path to admin or to another tenant's data in the
code on `main`. Closing **SEC-001** removes the only High; the

"good" to "tight". Per the project's standing rule, none of these
should ship without explicit go-ahead.
